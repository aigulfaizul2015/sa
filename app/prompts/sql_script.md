Задача
На основе предоставленной схемы базы данных сгенерировать production-ready SQL DDL скрипт создания таблиц.
————————————————————————————————
Критически важные правила
Использовать только входную схему
Разрешено:
- использовать только существующие таблицы;
- использовать только существующие поля;
- использовать только существующие типы данных;
- использовать только явно указанные PK/FK;
- использовать только явно указанные ограничения;
- использовать только явно существующие связи.

Запрещено:
- придумывать таблицы;
- придумывать поля;
- придумывать связи;
- придумывать ограничения;
- придумывать индексы;
- придумывать default values;
- додумывать бизнес-логику;
- переименовывать сущности;
- изменять naming;
- исправлять схему;
- оптимизировать схему;
- добавлять комментарии;
- добавлять пояснения;
- добавлять тестовые данные.

Если информация отсутствует:
- не додумывать;
- не генерировать элемент.
————————————————————————————————
Правила генерации SQL
Общие правила
- генерировать только SQL;
- использовать синтаксис указанной СУБД;
- соблюдать порядок зависимостей таблиц;
- сначала создавать master/reference таблицы;
- затем зависимые таблицы;
- FK создавать только после существования parent table.

Формат полей
Поля выводить строго в формате:

```sql
field_name DATA_TYPE
```
Пример:

```sql
id BIGINT PRIMARY KEY
hotel_id BIGINT NOT NULL
name VARCHAR(255)
created_at TIMESTAMP
```
Первичные ключи
Если PK указан:
- использовать PRIMARY KEY;
- composite PK создавать только если явно указано.
Внешние ключи
Создавать FK только если они явно указаны во входной схеме.
Пример:
```sql
CONSTRAINT fk_room_hotel
FOREIGN KEY (hotel_id)
REFERENCES hotel(id)
```
NULL / NOT NULL
- использовать только явно указанные ограничения;
- если nullable неизвестен — не придумывать NOT NULL.
Типы данных:
- адаптировать под указанную СУБД;
- не изменять смысл типов;
- не заменять типы без необходимости.
————————————————————————————————
Особенности СУБД
1. PostgreSQL
Для identity:
```sql
GENERATED BY DEFAULT AS IDENTITY
```
2. MySQL
Для identity:
```sql
AUTO_INCREMENT
```
3. MS SQL Server
Для identity:
```sql
IDENTITY(1,1)
```
4. Oracle
Использовать:
```sql
GENERATED BY DEFAULT AS IDENTITY
```
Sequence + trigger использовать только если явно требуется.
5. SQLite
Учитывать ограничения SQLite:
- отсутствует полноценный ALTER TABLE;
- типизация dynamic typing;
- использовать INTEGER PRIMARY KEY AUTOINCREMENT для автоинкремента;
- не использовать типы данных, отсутствующие в SQLite;
- VARCHAR преобразовывать в TEXT при необходимости;
- BOOLEAN хранить как INTEGER;
- TIMESTAMP хранить как TEXT / DATETIME только если явно указано.
Пример:
```sql
id INTEGER PRIMARY KEY AUTOINCREMENT
```
————————————————————————————————
Индексы
Создавать индексы только если они явно указаны во входной схеме.
Запрещено:
- автоматически добавлять индексы;
- оптимизировать производительность;
- создавать "рекомендуемые" индексы.
————————————————————————————————
Проверка перед выводом
Проверить:
- все таблицы существуют во входной схеме;
- все поля существуют во входной схеме;
- все FK существуют во входной схеме;
- отсутствуют выдуманные ограничения;
- отсутствуют выдуманные индексы;
- SQL соответствует указанной СУБД;
- отсутствует смешение синтаксиса разных СУБД.
