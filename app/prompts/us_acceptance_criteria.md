Задача
Сформировать критерии приемки (Acceptance Criteria) для каждой user story.
Критерии приемки должны:
- уточнять поведение системы (как детализация user requirements → functional level)
- быть проверяемыми
- быть однозначными
- покрывать позитивные и негативные сценарии
————————————————————————————————
Инструкции по выполнению
1. Проанализируй user story.
2. Определи ожидаемое поведение системы.
3. Определи условия выполнения.
4. Сформируй критерии приемки.
5. Проверь:
- проверяемость
- однозначность
- полноту
————————————————————————————————
Формат результата
Для каждой user story:
**US-<номер>**
**AC-<номер>:** Если <условие>, то <результат>.
————————————————————————————————
Требования к критериям
Каждый критерий:
- атомарный
- проверяемый
- конкретный
————————————————————————————————
ОБЯЗАТЕЛЬНО ВКЛЮЧИТЬ
Для каждой user story:
1. минимум 5-8 критериев
2. как минимум:
- 2 позитивный сценарий
- 2 негативный сценарий
- 1 граничный случай (если применимо)
————————————————————————————————
ЗАПРЕЩЕНО
- абстрактные формулировки
- “система должна корректно…”
- “валидные данные” без уточнения
- “и т.д.”
————————————————————————————————
Требования к детализации
Если используются данные: явно указывать поля и значения
————————————————————————————————
Ограничения
- Не переписывать user story
- Не описывать UI
- Не описывать архитектуру
- Не добавлять пояснения вне структуры
————————————————————————————————
Критерии качества
- Каждый критерий можно протестировать
- Нет неоднозначности
- Покрыты основные сценарии
- Нет дублирования
————————————————————————————————
Ограничение ответа
- Только список критериев приемки
- Без пояснений вне структуры
- Любой текст вне формата — ошибка
————————————————————————————————
Пример
**Пример**
**US-001. Как менеджер отеля, я хочу создавать бронирование, чтобы фиксировать заезды клиентов**
**AC-001:** Если пользователь заполняет поля "дата заезда", "дата выезда", "идентификатор клиента" и нажимает "создать", то бронирование сохраняется.
**AC-002:** Если поле "дата заезда" не заполнено, то система отображает ошибку и не сохраняет бронирование.
**AC-003:** Если дата выезда меньше даты заезда, то система отображает сообщение об ошибке.
**AC-004:** Если пользователь вводит данные, совпадающие с существующим бронированием (идентификатор клиента + даты), то система предупреждает о дублировании.
